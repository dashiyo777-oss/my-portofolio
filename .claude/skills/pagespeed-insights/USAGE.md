# PageSpeed Insights スキル 使い方ガイド

サイトの「表示速度の成績表」を、Claude が読んで日本語で解説してくれるようになる知識スキルです。
数値の意味・合格ライン・直し方を、聞くだけで教えてもらえます。

| 項目 | 内容 |
|---|---|
| スキル名 | `pagespeed-insights` |
| 種類 | 知識リファレンス（測定ツールではありません） |
| 保存場所 | `.claude/skills/pagespeed-insights/SKILL.md` |

---

## 01. これは何？

PageSpeed Insights（PSI）は、Google が提供する「Web ページの表示速度・使い心地」を測る無料ツールです。
ただ、出てくる結果は LCP・CLS・INP といった専門用語だらけで、数字を見ても**良いのか悪いのか分かりづらい**のが難点でした。

このスキルは、その PSI に関する知識を Claude に持たせるものです。
**ボタンを押すと自動で測定してくれる道具ではありません。** 測定結果を見せて質問すると、Claude が正しい基準・しきい値にもとづいて解説・アドバイスしてくれる「参考書」だと考えてください。

> **ひとことで言うと**
> 「この数字、合格してる？」「どこを直せばいい？」に、Claude が正確に答えられるようになります。

---

## 02. どんなときに動く？

特別な操作は不要です。会話の中で PSI や表示速度の話題が出ると、Claude が**自動でこのスキルを参照**します。次のような場面が「出番」です。

- PageSpeed Insights や Lighthouse の結果を読んでほしいとき
- Core Web Vitals に「合格 / 不合格」か判定してほしいとき
- ラボデータとフィールドデータが食い違う理由を知りたいとき
- スコアを上げる具体的な方法を相談したいとき

明示的に呼び出したいときは、チャットで `/pagespeed-insights` と入力しても OK です。

---

## 03. 使い方（3ステップ）

1. **PSI で自分のサイトを測る**
   [pagespeed.web.dev](https://pagespeed.web.dev/) を開き、あなたのポートフォリオの URL を入力して「分析」を押します。モバイルとデスクトップ両方の結果が出ます。

2. **結果を Claude に渡す**
   スコアや数値をコピーして貼り付けるか、結果画面のスクリーンショットを送ります。「このPSIの結果を見て」と一言添えるだけで十分です。

3. **質問する**
   「合格してる？」「一番のボトルネックは？」「どう直せばいい？」と聞けば、下の基準表にもとづいて Claude が解説と改善案を返します。

---

## 04. そのまま使える質問例

- この PSI の結果、Core Web Vitals に合格してる？
- LCP が 3.2 秒だった。これは良い？悪い？
- フィールドデータが「データ不足」で出ないのはなぜ？
- 実行するたびにスコアが変わるのはどうして？
- ポートフォリオのモバイルスコアを上げたい。優先順位は？

## 05. 合否ライン早見表

各指標の **75 パーセンタイル値** がこの基準で判定されます。
まず覚えるべきは **LCP・CLS・INP** の 3 つ（＝ Core Web Vitals）。この 3 つがすべて緑（Good）なら合格です。

| 指標 | 🟢 Good | 🟠 Needs Improvement | 🔴 Poor |
|---|---|---|---|
| **LCP**（表示の速さ） | 〜2.5秒 | 2.5〜4秒 | 4秒超 |
| **INP**（反応の速さ） | 〜200ms | 200〜500ms | 500ms超 |
| **CLS**（レイアウトのガタつき） | 〜0.1 | 0.1〜0.25 | 0.25超 |
| **FCP**（最初の描画） | 〜1.8秒 | 1.8〜3秒 | 3秒超 |
| **TTFB**（サーバー応答・実験的） | 〜800ms | 0.8〜1.8秒 | 1.8秒超 |

判定ルールの詳細は [`SKILL.md`](SKILL.md) の「Core Web Vitals assessment」を参照してください。

## 07. 覚えておくと得するコツ

- **スコアは毎回少し変わる** — ネットワークや測定サーバーの状態で数点ぶれます。1回の数字に一喜一憂せず、傾向で見ましょう。
- **ラボが満点でも安心しない** — Lighthouse（ラボ）で90点でも、実ユーザー（フィールド）の体感が良いとは限りません。最終的にはフィールドデータが本番です。
- **まずモバイルを見る** — 判定は厳しめのモバイル条件（中位機・低速回線）が基準。改善はモバイルから手をつけるのが効率的です。
